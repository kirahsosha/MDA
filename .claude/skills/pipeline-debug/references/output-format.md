# Pipeline 调试输出格式

按以下结构组织响应。若用户只要求快速判断，可以压缩为摘要 + 高优先级问题 + 建议。

````markdown
## 分析摘要

- 节点总数：X
- 入口节点：...
- 发现问题：Y
- 高风险问题：Z

## 节点语义分析

| 节点名              | 类型 | 域      | 角色            | 预期行为         |
| ------------------- | ---- | ------- | --------------- | ---------------- |
| ExampleEnterPage    | 动作 | Example | Enter<Page>     | 点击进入目标页面 |
| ExamplePageEntered  | 检测 | Example | Entered         | 确认页面已进入   |
| CommonConfirmDialog | 动作 | Common  | Confirm<Object> | 确认通用弹窗     |

## 发现的问题

### 严重问题

1. [问题描述] - [节点名] - [为什么会失败]

### 警告

1. [警告描述] - [节点名] - [潜在影响]

### 命名与代码不匹配

1. [节点名] - [名称预期行为] - [代码实际行为]

### 优化建议

1. [建议] - [收益]

## 修正后的 Pipeline

```json
{
    "...": {}
}
```

## 改动说明

1. [改动 1]：[原因]
2. [改动 2]：[原因]

## 测试建议

- [如何用截图、日志或节点测试验证修复]
- [需要覆盖的正常路径和异常路径]
````

## 输出原则

- 先报告会导致执行失败、误点、卡死的正确性问题。
- 性能与风格建议放在后面，不要淹没根因。
- 如果无法确认，写“需要运行日志/截图验证”，不要把推测当结论。
- 修正片段只包含相关节点，除非用户要求完整文件。
